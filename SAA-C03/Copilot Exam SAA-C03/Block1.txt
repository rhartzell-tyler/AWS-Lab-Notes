Question 8  
A company runs a high-traffic web application using Amazon API Gateway (REST API) and AWS Lambda. The application experiences sudden traffic spikes that cause throttling errors (429) from API Gateway. The company wants to handle these spikes gracefully without significantly overprovisioning limits. What should a solutions architect recommend?

A. Increase the account-level API Gateway throttle limits to the maximum allowed.
B. Configure usage plans and API keys to limit client traffic.
C. Place an Amazon CloudFront distribution in front of API Gateway and enable caching for idempotent GET requests.
D. Add an Amazon SQS queue between API Gateway and Lambda to buffer requests.

Question 8 — Correct Answer: C
Your answer: D

Why C is correct
Putting CloudFront in front of API Gateway is a real AWS exam pattern for handling sudden traffic spikes:

CloudFront absorbs bursts

CloudFront caches GET responses

CloudFront reduces direct load on API Gateway

CloudFront smooths out request rates

This is exactly what AWS recommends for REST APIs experiencing throttling.

Why D is wrong
You cannot put SQS between API Gateway and Lambda for a synchronous REST API.
API Gateway expects an immediate response — SQS breaks that contract.

Question 9  
A company is migrating an on-premises monolithic application to AWS. The application currently uses a shared file system for session data and user uploads. The company wants to move to a microservices architecture using Amazon ECS on Fargate and an Application Load Balancer. Session stickiness is not desired; any task should be able to serve any request. Which architecture best supports this requirement?

A. Store session data and user uploads on an Amazon EFS file system mounted to all tasks.
B. Store session data in Amazon ElastiCache for Redis and user uploads in Amazon S3.
C. Use EBS volumes attached to each task and replicate data between them.
D. Use DynamoDB for both session data and file storage.

Question 9 — Correct Answer: B
Your answer: D

Why B is correct
For microservices with stateless tasks:

Session data → ElastiCache Redis (low latency, shared, scalable)

User uploads → S3 (durable, scalable, shared)

This is the standard AWS reference architecture.

Why D is wrong
DynamoDB is great, but:

It’s not appropriate for file storage

It’s not ideal for session storage requiring sub‑millisecond latency

It’s not used for binary uploads

Question 13  
A company runs a latency-sensitive trading application on Amazon EC2 instances in a single Availability Zone. The application connects to an Amazon RDS for MySQL database in the same AZ. The company is concerned about AZ-level failures but also wants to minimize latency between the application and the database. Which solution provides the best balance between high availability and low latency?

A. Enable Multi-AZ on the RDS instance and deploy EC2 instances across multiple AZs behind a Network Load Balancer.
B. Enable Multi-AZ on the RDS instance and keep EC2 instances in a single AZ, using Auto Scaling within that AZ.
C. Use Amazon Aurora MySQL with a reader in another AZ and direct all reads to the reader.
D. Deploy separate RDS instances in each AZ and use application logic to choose the closest database.

Question 13 — Correct Answer: B
Your answer: C

Why B is correct
The company wants:

Low latency (keep EC2 in one AZ)

High availability (RDS Multi‑AZ)

This is the exact AWS‑recommended pattern for latency‑sensitive workloads.

Why C is wrong
Aurora with a reader in another AZ helps with read scaling, not write availability.
Also, the question is about latency between app and DB, not read scaling.
